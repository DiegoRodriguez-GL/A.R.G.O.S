# Empirical evaluation of ARGOS

This document describes the methodology, ground truth and canonical
results of the ARGOS empirical evaluation framework (Module 7, OE6 of
the project objectives). It is the public companion of the
`argos-eval` package and the `argos eval` CLI verb.

## Goal

The framework answers two operational questions:

1. **Construct validity.** Given an agent that is *vulnerable by
   construction* to a known set of OWASP ASI threats, does ARGOS
   detect every one of them?
2. **Specificity.** Given a *hardened* version of the same agent,
   does ARGOS abstain from raising findings on the same set of
   probes?

A passing benchmark is therefore **a 100% diagonal confusion matrix**:
true positives where the lab declares vulnerability; true negatives
everywhere else. Any off-diagonal cell is a defect either in a probe
detector or in the lab agent.

## Design

### Lab agents

Six deterministic agents (Phase 2 of Module 7) implement the
[`AgentTransport`](../packages/argos-redteam/src/argos_redteam/transport.py)
interface. Each agent is a pure function from a transcript to an
assistant reply: no LLM sampling, no temperature, no network. This
forfeits external validity (which is recovered separately, see
"Transferability" below) in exchange for a fully reproducible
benchmark whose confusion matrix is byte-stable across runs.

| Scenario | Vulnerable variant | Hardened variant |
|----------|-------------------|------------------|
| ReAct + MCP tools | `lab.react.vulnerable` | `lab.react.hardened` |
| LangGraph supervisor-worker | `lab.langgraph.vulnerable` | `lab.langgraph.hardened` |
| Memory + RAG | `lab.memory.vulnerable` | `lab.memory.hardened` |

Each variant declares a `vulnerability_surface` ClassVar listing the
probe IDs the variant is meaningful against. The hardened variants
share the surface of their vulnerable counterpart but block instead
of fire on every member of it.

### Ground truth

The matrix `(agent_id, probe_id) -> expected_outcome` lives in
[`packages/argos-eval/src/argos_eval/data/ground_truth.yaml`](../packages/argos-eval/src/argos_eval/data/ground_truth.yaml).
Its compact form lists only the cells expected to **fire**; every
other cell defaults to **block**. The file is generated from the
declared surfaces of the lab agents but can be overridden by hand
without touching Python; the runner uses the YAML, not the surface
declarations, as the source of truth.

### Probes

The full ARGOS red-team catalogue (20 probes across OWASP ASI01-ASI10,
two probes per top-level category) is exercised against every agent.

### Metrics

The framework computes the five canonical binary-classification metrics
plus auxiliaries:

| Metric | Definition |
|--------|------------|
| Precision | TP / (TP + FP) |
| Recall (sensitivity) | TP / (TP + FN) |
| Specificity | TN / (TN + FP) |
| Accuracy | (TP + TN) / total |
| F1 score | harmonic mean of precision and recall |
| MCC (Matthews) | a single-number summary in [-1, +1], robust to class imbalance |

Confidence intervals over proportions use Wilson's formula
(Wilson 1927; Newcombe 1998). Bootstrap resampling (Efron 1979) is
available for arbitrary statistics through
`argos_eval.bootstrap_ci`.

## Reproducing the canonical run

### Quick command

```bash
argos eval --output argos-eval.html --json argos-eval.json
```

### Stand-alone script

```bash
uv run python scripts/canonical_eval.py
```

### Programmatic

```python
from argos_eval import all_agents, default_ground_truth, run_suite

report = run_suite(
    list(all_agents()),
    list(__import__("argos_redteam").all_probes()),
    default_ground_truth(),
)
```

### By OWASP ASI category

For every top-level ASI category the breakdown is uniform: TP=2,
TN=10, FP=FN=0. Two probes per category contribute, both fired by
the relevant vulnerable agent and blocked by the four other agents.

### By agent

| Agent | TP | FP | TN | FN |
|-------|----|----|----|----|
| `lab.react.vulnerable` | 6 | 0 | 14 | 0 |
| `lab.react.hardened` | 0 | 0 | 20 | 0 |
| `lab.langgraph.vulnerable` | 6 | 0 | 14 | 0 |
| `lab.langgraph.hardened` | 0 | 0 | 20 | 0 |
| `lab.memory.vulnerable` | 8 | 0 | 12 | 0 |
| `lab.memory.hardened` | 0 | 0 | 20 | 0 |

The vulnerable variants accumulate exactly their declared surface
size (6 for ReAct, 6 for LangGraph, 8 for Memory) as TP. The hardened
variants are pure TN: every probe blocks. Both variants share the same
total trial count (20) because the runner exercises the full probe
catalogue against every agent.

## Threats to validity

Following the structure of Wohlin et al. (2012):

- **Construct validity.** The lab agents are vulnerable *by
  construction*: their scripts are designed to match the patterns the
  probes detect. This guarantees the benchmark exercises the
  detection surface, not unrelated capability. Mitigation: each agent
  surface covers at least three OWASP ASI top-level categories so the
  framework cannot trivially over-fit a single threat class.
- **Internal validity.** Determinism is total: pure functions, no
  network, no temperature. Two runs against the same lab produce
  byte-identical case lists. The Wilson confidence intervals reported
  in the global table reflect the binomial sampling uncertainty under
  the operational definition (one trial per (agent, probe) pair), not
  any stochasticity in the agents.
- **External validity.** The benchmark does not, by itself, prove
  that ARGOS performs identically on real LLM endpoints. The
  transferability annex (below) addresses the generalisation question
  separately so it cannot contaminate the construct-validity result.
- **Statistical conclusion validity.** Wilson intervals are
  preferred to the textbook normal approximation because they
  preserve the [0, 1] domain even at extreme proportions; bootstrap
  is preferred for non-proportion statistics (e.g. macro-averaged
  F1 across categories).

## Transferability (future work)

A separate annex covers transferability to real LLM endpoints. The
roadmap, when API budget and access are available:

1. Provide a thin `LLMTransport` adapter (subclass of
   `argos_redteam.HttpTransport`) wired to the OpenAI / Anthropic
   chat-completions schema, with a `--max-requests` denial-of-wallet
   cap (already exposed in `argos redteam`).
2. Run a small subset (5-10 probes against 1-2 models, 3-5 samples
   each at low temperature) and compare per-probe positive rates
   against the lab numbers.
3. Discuss disagreements as either *probe brittleness* (real model
   refused where the lab agent fired) or *agent over-fitting* (lab
   reply matches the detector regex too cleanly).

The annex is **out of scope of the lab benchmark**: its purpose is
to estimate the gap between lab construct validity and real-world
behaviour, not to score ARGOS itself.
